# Serialization order and stable identity

Bytecode IDs are positions, while expression addresses are identities. Keeping that distinction is essential because several compiler collections are unordered.

## Numbering and emission

[`compile()`](../../../src/compiler/compiler.cpp) numbers twice: once before constructing `GraphLinker`, then again after linking. Each expression implementation must serialize in its numbering order:

- unary: child, node;
- binary: left subtree, right subtree, node;
- block: block, statement subtrees;
- if: condition, if, then block, optional else/else block, merge;
- function: function, body.

`countInstructions()` must equal the number of lines emitted by `toByteCode()` excluding optional debug comments.

## Why unordered dependent positions are not identity

`Expression::dependents` is an `unordered_set<ExprDependent>`. Iteration order is unspecified and may change after insertion, rehashing, allocation changes, or a different standard-library implementation. [`UnaryCallExpression::depRemaps`](../../../src/compiler/expression.hpp) therefore keys remaps by `Expression *`, not an integer position.

During [`UnaryCallExpression::toByteCode()`](../../../src/compiler/expression.cpp), serialization iterates the actual `dependents` set, tracks the emitted index, looks up each destination pointer in `depRemaps`, and only then writes that dependent index. This binds a remap to the same dependent ordering actually written by `Expression::toByteCode()`.

Subprogram-relative targets use expression IDs plus `subprogramOffset`. Resolution is delayed until final IDs exist. This is the same general rule: retain object identity during graph construction; derive positional indices at the serialization boundary.

## Other unordered structures

Function use summaries and resource tables are unordered maps. Argument-remap sections derived from parameter `firstUses` are emitted through an unordered map, and unordered analysis state can also affect dependent insertion/emission order; meaning must not depend on either textual order. `GraphLinker::expressions` is intentionally `std::map<int,...>` because traversal semantics do depend on ID order. Branch lookups are keyed by explicit instruction IDs rather than vector positions.

`ExprDependent` hashes the destination address, while equality compares destination address and argument index. Equal entries must hash consistently; distinct argument slots for the same destination remain distinct edges.

## Concrete observed checks

I ran:

```sh
build/Tests --gtest_filter='ExprDependent.*:linkGraph.setsCallDepRemaps:compile.compilesCompoundExpressions'
```

Observed: all selected tests passed. The hasher tests verified deduplication for equal destination/index pairs and separation for different pairs. `compile.compilesCompoundExpressions` verified stable recursive IDs for `1 + 1 + 1` and this exact five-line shape:

```text
0 4.0 1 1
0 3.0 1 1
0 3.1 1 1
2 4.1 6
2  6
```

## Invariants and change hazards

- Never store an unordered-container iteration index for later use.
- Do not use expression IDs as long-lived keys before final numbering.
- Resolve call remap dependent indices in the same loop/order used to emit dependents.
- If deterministic byte-for-byte output becomes a requirement, explicitly sort semantic entries before both relevant emissions; do not sort only one side of an index relationship.
- Tests should parse generated bytecode and validate pointer relationships, not depend solely on textual order where order has no semantic meaning.
