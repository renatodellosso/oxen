# Bytecode format

Bytecode is line-oriented text generated by [`Expression::toByteCode()`](../../../src/compiler/expression.cpp) and parsed by [`BytecodeParser`](../../../src/interpreter/bytecodeParser.cpp). There is currently no version header.

## Common line prefix

Every instruction line has:

```text
<incoming-dependency-count> <outgoing-dependent-list> <opcode> [arguments...]
```

The first field is how many dependency signals this instruction must receive. The comma-separated list describes where this instruction sends its result; each item is `<instruction-id>` or `<instruction-id>.<argument-index>`. An empty dependent list produces two adjacent spaces before the opcode. IDs are zero-based instruction positions after comment lines have been discarded.

Example:

```text
0 2.0 1 1
0 2.1 1 2
2  6
```

Instruction 0 is literal `1` and supplies argument 0 of instruction 2. Instruction 1 supplies argument 1. Opcode 6 (`Add`) waits for two dependencies and has no outgoing dependents.

## Opcodes

The numeric ABI is the declaration order in [`InstructionType`](../../../src/instruction.hpp):

| ID | Type | ID | Type | ID | Type |
|---:|---|---:|---|---:|---|
| 0 | Block | 9 | Divide | 18 | GoTo |
| 1 | GetLiteral | 10 | Negate | 19 | Print |
| 2 | ReferenceIdentifier | 11 | CompareEquals | 20 | Function |
| 3 | GetIdentifier | 12 | CompareLessThan | 21 | Call |
| 4 | Declare | 13 | CompareLessThanEquals | 22 | CompareNotEquals |
| 5 | Set | 14 | CompareGreaterThan | 23 | Else |
| 6 | Add | 15 | CompareGreaterThanEquals | 24 | BranchMerge |
| 7 | Subtract | 16 | If | 25 | Return |
| 8 | Multiply | 17 | While |  |  |

Changing enum order changes the file format.

## Arguments

- Root expressions append their token raw value. This covers literals, identifiers, and the relative integer distance on `GoTo`.
- `Block` appends `<nested-instruction-count> <direct-call-count> <call-offset>...`. The first count excludes the block instruction. The call entries are elements of that block's `expressions` whose type is exactly `Call`; their offsets are relative to the block ID. In practice a `CallExpression` is itself a block whose first child is that unary `Call` instruction.
- `Function` appends `<return-type> <name> <generated-loop-body-bool>`. Its body block follows immediately.
- `Call` is a unary expression, so its function-name `GetIdentifier` line precedes the call line. The call line then appends four counted sections: dependent-index remaps to callee-relative instruction IDs, argument-expression offsets remapped to parameter first uses, parameter declaration offsets, and callee-relative return instruction IDs. The exact layout is documented above `UnaryCallExpression::toByteCode()`.
- Arithmetic, comparison, `Declare`, `Set`, `If`, `While`, `Print`, `Else`, `BranchMerge`, and `Return` take values through dependency arguments rather than extra bytecode fields.

`--debug` causes expressions to emit `//` comment lines containing `toString()`. `BytecodeParser::buildSingleInstruction()` discards a line whose first field is `//`, so comments do not consume instruction IDs.

`buildArg()` recognizes quoted strings, exact `true`/`false`, integer text, and otherwise identifiers. It retains spaces inside strings but strips their surrounding quotes; it does not decode escapes and treats the next `"` as the terminator. The parser currently uses `atoi`, unchecked instruction indexing, and enum casts without a format/version validation layer; bytecode should be treated as trusted compiler output.

## Concrete observed parse

The following command verifies the compiler and parser examples:

```sh
build/Tests --gtest_filter='compile.compilesBasicProgram:buildInstructions.buildsSingleInstruction:buildInstructions.buildsCompoundInstructions'
```

Observed: all 3 passed. `buildsSingleInstruction` parsed the example above into three instructions, resolved both dependent pointers to instruction 2, preserved argument indices 0/1, and set the `Add` dependency count to 2.

## Change hazards

Keep writer and reader changes atomic. Give new arguments unambiguous spacing
and string rules. For a new instruction, append an enum value where possible,
add compiler emission and executor dispatch, preserve parser-compatible
arguments, and add malformed-bytecode tests. If external compatibility becomes
important, add a format version before changing existing opcode values or
argument layouts. Verify the text in
[`tests/compiler/compiler.cpp`](../../../tests/compiler/compiler.cpp), pointer
resolution in
[`tests/interpreter/bytecodeParser.cpp`](../../../tests/interpreter/bytecodeParser.cpp),
and execution in [`tests/interpreter/executor.cpp`](../../../tests/interpreter/executor.cpp)
or the E2E suite.
